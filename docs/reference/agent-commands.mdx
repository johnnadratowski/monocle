title: "Agent Commands"
description: "CLI commands for agents to interact with a running monocle session."
---

These commands are used by agents â€” via installed skills, MCP tools, or directly from the command line â€” to communicate with a running monocle session.

When registered in **MCP tools mode**, agents call these operations as MCP tools (`review_status`, `get_feedback`, `send_artifact`, `add_files`, `remove_files`, `set_file_groups`, `add_annotations`, `set_review_name`, `set_base_ref`) instead of CLI commands. The behavior is identical â€” the MCP server connects to the same Unix socket internally.

When registered in **skills mode**, agents invoke the corresponding CLI commands below.

<Note>
  When the agent's working directory differs from monocle's, use `--workdir` (`-C`) to point commands at the correct repository. Alternatively, set `MONOCLE_SOCKET` or `MONOCLE_WORKDIR` to achieve the same effect via environment variables. Without one of these, commands look for a socket based on the current directory and may not find a running session. Press `I` in the monocle TUI to see the active socket path.
</Note>

## `monocle review status`

Check the current review status of the running monocle session.

```bash
monocle review status [--json] [--workdir PATH] [--socket PATH]
```

### Flags

| Flag | Short | Description |
|------|-------|-------------|
| `--json` | | Output the result as structured JSON instead of plain text. |
| `--workdir PATH` | `-C` | Override the working directory. Also reads from `MONOCLE_WORKDIR`. |
| `--socket PATH` | | Override the Unix socket path. Also reads from `MONOCLE_SOCKET`. |

### Output

Without `--json`, prints a status string or summary. The status reflects whether feedback is pending, a pause has been requested, or the session is idle.

### MCP tool equivalent

In MCP tools mode, call the `review_status` tool with no parameters.

## `monocle review get-feedback`

Retrieve pending review feedback from the queue.

```bash
monocle review get-feedback [--wait] [--json] [--workdir PATH] [--socket PATH]
```

When the reviewer submits a review, it is placed in a queue. Multiple queued reviews are combined and delivered together as a single response.

Without `--wait`, the command returns immediately. If no feedback is pending it prints `No feedback pending.` and exits.

### Flags

| Flag | Short | Description |
|------|-------|-------------|
| `--wait` | | Block until feedback is available. The command does not return until the reviewer submits a review. Used by the `/review-plan-wait` skill. |
| `--json` | | Output the result as structured JSON. |
| `--workdir PATH` | `-C` | Override the working directory. Also reads from `MONOCLE_WORKDIR`. |
| `--socket PATH` | | Override the Unix socket path. Also reads from `MONOCLE_SOCKET`. |

### MCP tool equivalent

In MCP tools mode, call the `get_feedback` tool:

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `wait` | boolean | No | When `true`, blocks until feedback is available. |

## `monocle review send-artifact`

Send content to the reviewer â€” a plan, architecture document, summary, or any other text â€” so it appears in the monocle sidebar alongside file diffs.

```bash
monocle review send-artifact --title TEXT [--file PATH] [--id ID] [--type EXT] [--wait] [--json] [--workdir PATH] [--socket PATH]
```

If you run the command again with the same `--id`, monocle updates the existing entry and shows a diff between the old and new versions.

When `--file` is not provided, the command reads content from stdin.

### Flags

| Flag | Short | Description |
|------|-------|-------------|
| `--title TEXT` | | **(Required)** Title displayed in the monocle sidebar. |
| `--file PATH` | | Path to the file to submit. When omitted, reads from stdin. |
| `--id ID` | | Identifier for the artifact. Use the same ID on subsequent calls to update the content in place. Defaults to the filename when `--file` is used. |
| `--type EXT` | | File extension used for syntax highlighting (e.g. `md`, `go`, `py`, `ts`). |
| `--wait` | | Block until the reviewer responds with feedback. Used by the `/review-plan-wait` skill. |
| `--json` | | Output the result as structured JSON. |
| `--workdir PATH` | `-C` | Override the working directory. Also reads from `MONOCLE_WORKDIR`. |
| `--socket PATH` | | Override the Unix socket path. Also reads from `MONOCLE_SOCKET`. |

### Reading from stdin

```bash
echo "## Plan\n- Step 1\n- Step 2" | monocle review send-artifact --title "My plan" --type md
```

### Reading from a file

```bash
monocle review send-artifact --title "Architecture" --file docs/architecture.md --id architecture.md
```

### MCP tool equivalent

When using MCP tools mode, the `send_artifact` tool accepts these parameters:

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `id` | string | No | Identifier for the artifact. Use the same ID on subsequent calls to update the content in place. Defaults to the filename when `file_path` is provided. |
| `title` | string | Yes | Title displayed in the monocle sidebar. |
| `content` | string | No | The artifact content. Either `content` or `file_path` must be provided. |
| `file_path` | string | No | Absolute path to a file to read as the artifact content. If `id` is empty, defaults to the filename. |
| `content_type` | string | No | File extension for syntax highlighting (e.g. `md`, `go`, `py`). |

## `monocle review add-files`

Add one or more files or directories to the current review session so they appear in the monocle diff viewer.

```bash
monocle review add-files <paths...> [--json] [--workdir PATH] [--socket PATH]
```

### Arguments

| Argument | Description |
|----------|-------------|
| `paths` | **(Required)** One or more file or directory paths to add. Paths are resolved to absolute paths before being sent to monocle. |

### Flags

| Flag | Short | Description |
|------|-------|-------------|
| `--json` | | Output the result as structured JSON. |
| `--workdir PATH` | `-C` | Override the working directory. Also reads from `MONOCLE_WORKDIR`. |
| `--socket PATH` | | Override the Unix socket path. Also reads from `MONOCLE_SOCKET`. |

### MCP tool equivalent

In MCP tools mode, call the `add_files` tool:

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `paths` | string array | Yes | File or directory paths to add to the session. |

### Example

```bash
monocle review add-files src/feature.go tests/
```

## `monocle review group-files`

Assign category, group, and ordering metadata to the changed files so the reviewer can read the change as a top-down story in the **grouped sidebar view** (press `f` to cycle to it). Provide a JSON manifest of entries; each entry's `path` must match a changed file from `monocle review status`.

```bash
monocle review group-files [--file manifest.json] [--replace] [--json] [--workdir PATH] [--socket PATH]
```

The manifest is a JSON array of entries (or an object with an `entries` array), read from `--file` or stdin (`-`):

```json
[
  { "path": "ui/App.tsx",      "group": "UI",       "group_order": 0, "sort_index": 0 },
  { "path": "api/handler.go",  "group": "Backend",  "group_order": 1 },
  { "path": "db/schema.sql",   "group": "Database", "group_order": 2, "criticality": 5 }
]
```

When a change spans multiple independent efforts, add an optional `workstream` (with `workstream_order`) to nest groups under a top-level workstream header â€” files then read as **workstream â†’ group â†’ files**:

```json
[
  { "path": "search/ui.tsx",    "workstream": "Search",   "workstream_order": 0, "group": "UI",      "group_order": 0 },
  { "path": "search/api.go",    "workstream": "Search",   "workstream_order": 0, "group": "Backend", "group_order": 1 },
  { "path": "payments/api.go",  "workstream": "Payments", "workstream_order": 1, "group": "Backend", "group_order": 0 }
]
```

### Entry fields

| Field | Type | Description |
|-------|------|-------------|
| `path` | string | **(Required)** Changed-file path, as shown by `review status` (repo-relative). |
| `workstream` | string | Optional top-level grouping above `group`. Use it when the change spans multiple independent efforts; omit it for a single-workstream change. Files nest as workstream â†’ group â†’ files. |
| `workstream_order` | int | Display order of the workstream; lower workstreams are shown first. |
| `group` | string | Free-form group label (e.g. `UI`, `Backend`, `Database`). Files without a group fall back to a heuristic category. |
| `group_order` | int | Display order of the group; lower groups are shown first. |
| `sort_index` | int | Order within the group; lower first. Defaults to Monocle's computed import order (dependencies first), then lines-changed. |
| `category` | string | Override the heuristic category: `code`, `test`, `config`, `docs`, or `build` (drives the icon). |
| `criticality` | int | Optional importance hint (higher = more critical). |

### Flags

| Flag | Short | Description |
|------|-------|-------------|
| `--file PATH` | | Path to the JSON manifest (`-` or omitted reads stdin). |
| `--replace` | | Replace all existing grouping metadata instead of merging. |
| `--json` | | Output the result as structured JSON. |
| `--workdir PATH` | `-C` | Override the working directory. Also reads from `MONOCLE_WORKDIR`. |
| `--socket PATH` | | Override the Unix socket path. Also reads from `MONOCLE_SOCKET`. |

### MCP tool equivalent

In MCP tools mode, call the `set_file_groups` tool:

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `entries` | object array | Yes | Grouping entries (`path`, `workstream`, `workstream_order`, `group`, `group_order`, `sort_index`, `category`, `criticality`). |
| `replace` | boolean | No | Replace all prior grouping metadata. |

### Example

```bash
monocle review group-files --file groups.json --replace
```

## `monocle review annotate`

Attach short rationale notes to specific ranges of the changed code to guide the reviewer, each linking out to the docs that motivate it. **Annotations are shown to the reviewer only â€” they are not review feedback and are never sent back to the agent.** They render as a distinct box below the code range, with a colored bar marking the annotated lines.

```bash
monocle review annotate [--file manifest.json] [--replace] [--json] [--workdir PATH] [--socket PATH]
```

The manifest is a JSON array of entries (or an object with an `entries` array), read from `--file` or stdin:

```json
[
  {
    "file": "api/handler.go",
    "line_start": 40,
    "line_end": 52,
    "summary": "Guards the on-chain deposit before the goal is created",
    "refs": [
      { "kind": "file", "doc": "docs/todo.md", "label": "srv-008", "start_line": 120, "end_line": 135 }
    ]
  }
]
```

### Entry fields

| Field | Type | Description |
|-------|------|-------------|
| `file` | string | **(Required)** Changed-file path (from `review status`) the note explains. |
| `line_start` / `line_end` | int | **(Required)** The exact code range the note covers. |
| `summary` | string | One line: why this code exists / what it does. |
| `refs` | array | Doc links. Each has `kind` (`file` for a repo path, `artifact` for a sent artifact id), `doc` (path or id), `label` (link text), and a `start_line`/`start_col`/`end_line`/`end_col` range into the doc (1-based lines, 0-based cols). |

### Flags

| Flag | Short | Description |
|------|-------|-------------|
| `--file PATH` | | Path to the JSON manifest (`-` or omitted reads stdin). |
| `--replace` | | Clear all existing annotations first (default replaces per file). |
| `--json` | | Output the result as structured JSON. |
| `--workdir PATH` | `-C` | Override the working directory. Also reads from `MONOCLE_WORKDIR`. |
| `--socket PATH` | | Override the Unix socket path. Also reads from `MONOCLE_SOCKET`. |

### Validation and response

Each entry is validated before it is stored. The response reports how many were **accepted** and lists any **rejected** entries with a reason so you can correct and resend. An entry is rejected when:

- `line_start` is less than 1,
- `line_end` is less than `line_start`,
- `summary` is empty, or
- `file` is not one of the review's changed files.

Doc refs that can't be resolved (a wrong repo path or unknown artifact id) don't reject the entry â€” they come back as **warnings**. The annotation is still stored, but the link will read "not found" to the reviewer, so fix the ref and resend.

Annotations are pinned to a round's exact code ranges, so they are **cleared automatically when the reviewer submits and the round advances**. Re-annotate against the revised code each round.

### MCP tool equivalent

In MCP tools mode, call the `add_annotations` tool with the same `entries` (and optional `replace`) shape. The tool returns the same accepted / rejected / warning summary as text.

### Example

```bash
monocle review annotate --file annotations.json
```

## `monocle review set-name`

Start and name a review. The name (shown in Monocle's top bar) also marks the review as **in progress** â€” naming a review is how you begin one. Call it once at the start of each review.

```bash
monocle review set-name "Add OAuth login" [--force] [--json] [--workdir PATH] [--socket PATH]
```

Lifecycle rules:

- The **name is required and must be non-empty**. Only the reviewer closes a review (by approving or clearing it) â€” the agent can't.
- Calling it again with the **same name** is a no-op.
- Calling it with a **new name while a review is already open** replaces the old review automatically **if the reviewer hasn't left any comments**. If the reviewer **has** commented, the call is **refused** to protect their work â€” pass `--force` to discard the in-progress review and start the new one.
- Approving or clearing a review resets its name (and the diff base), so the next review starts clean.

### Flags

| Flag | Description |
|------|-------------|
| `--force` | Discard an in-progress review that already has reviewer comments and start this one. |

### MCP tool equivalent

In MCP tools mode, call the `set_review_name` tool with `name` (required) and optional `force` (boolean).

## `monocle review set-base-ref`

Review work you have **already committed**. By default monocle only shows uncommitted working-tree changes, so once your changes are committed they fall behind the base and disappear from the diff. This command points the diff at a base commit so everything since it â€” including your commits â€” is shown for review.

```bash
monocle review set-base-ref <ref> [--json] [--workdir PATH] [--socket PATH]
monocle review set-base-ref --reset
```

The `<ref>` is the commit to diff **against** â€” its own changes are excluded and the diff shows `ref..worktree`. Pass the branch you started from, a SHA, or `HEAD~N`. The base reverts to `HEAD` automatically once the reviewer submits, so the ref is scoped to a single review.

### Arguments

| Argument | Description |
|----------|-------------|
| `ref` | The commit to diff against (e.g. `main`, `HEAD~3`, a SHA). Required unless `--reset` is given. |

### Flags

| Flag | Short | Description |
|------|-------|-------------|
| `--reset` | | Revert to working-tree review immediately (re-enable auto-advance to `HEAD`). |
| `--json` | | Output the result as structured JSON. |
| `--workdir PATH` | `-C` | Override the working directory. Also reads from `MONOCLE_WORKDIR`. |
| `--socket PATH` | | Override the Unix socket path. Also reads from `MONOCLE_SOCKET`. |

### MCP tool equivalent

In MCP tools mode, call the `set_base_ref` tool:

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `ref` | string | Yes* | Commit to diff against (its own changes excluded). |
| `reset` | boolean | No | When `true`, revert to working-tree review. *Provide either `ref` or `reset`, not both. |

### Examples

```bash
# Review every commit made on this branch since it forked from main
monocle review set-base-ref main

# Review the last 3 commits
monocle review set-base-ref HEAD~3

# Go back to reviewing uncommitted working-tree changes
monocle review set-base-ref --reset
```

<Note>
The base ref is scoped to one review. After the reviewer submits (approve or request changes), monocle automatically returns to working-tree review against `HEAD`. Re-run the command for the next round if you want committed work reviewed again.
</Note>
